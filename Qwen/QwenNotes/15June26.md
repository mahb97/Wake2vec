### session Qs

3B P2 walled at val 5.33 within 100 steps and held it for six consecutive evals. Qwen P2 dropped to 13.43 at step 100 so the question is what happens next:

- **keeps descending** (13.43 to 12.x to 11.x): the LoRA ceiling is scale-dependent, 3B-specific. Qwen's 14B scale + WakeOverlay-then-LoRA architecture gives it headroom 3B didn't have. 
- **walls now** (13.43 to 13.4 to 13.4): Qwen walls at a lower val than 3B, universal-but-scale-shifted. 
- **either way**: the -1.62 first drop already exceeds 3B's entire pre-wall LoRA extraction (-1.35). Qwen P2 has pulled more from LoRA than 3B managed before walling.

### Speed note

P2 is running ~98s/step, ~40% faster than P1's final ~168s/step. counterintuitive but real as P1 (embedding-only) trained 224M Wake-row params, which really consumed GPU memory and forced more layers onto CPU, so every step round-tripped more data across the bus. P2 (LoRA-only) trains ~10-50M adapter params with the embedding matrix frozen, freeing GPU memory so fewer layers offload. on memory-constrained hardware, the "lightweight" embedding phase is actually heavier than LoRA.

val **8.27** at step 150. down 5.16 points from step 100's 13.43, in a single 50-step interval which makes it the largest single-eval val drop anywhere in the entire wake2vec project, across all models and all phases.

**the scale-dependent-ceiling reading is now strongly confirmed.** shows that the LoRA wall that the 128K-vocab Llamas hit is NOT universal. at 14B scale with the WakeOverlay-then-LoRA architecture, LoRA finds enormous descent where 3B found a brick wall in 100 steps. this is the cleanest possible refutation of the argument that "the LoRA ceiling is architecture-wide."

where does it stop? currently unknown. at this rate it could blow through the 5.33 territory 3B walled at. more likely it decelerates somewhere but "somewhere" is for now an open question. EVAL_STEPS=50 means the next few evals (200, 250, 300) map the descent curve in detail.
