## Llama 3.1-8B P2 session 1 (launch)

8B P2 is live. the fourth Llama-family P2 (after TinyLlama, Llama 1B, Llama 3B), and the one positioned to answer the question the lineup has been building toward: does 8B LoRA wall like 3B (val 5.33 ceiling) or descend like Qwen (broke the wall at 14B)?

### Launch config

| Param | Value |
|-------|-------|
| Model | meta-llama/Llama-3.1-8B (4-bit NF4) |
| P1 source | step 1200 (best val 11.3603, compositional-init embeddings) |
| Embeddings | FROZEN at P1 best-val |
| LoRA | r=8, α=16, dropout=0.1, targets q/k/v/gate/up/down |
| LR | 2e-5 |
| **SEQ_LEN** | **512 (matches 3B P2 for direct comparability)** |
| Batch | 1 x 16 = 16 effective |
| Max steps | 3000 (full P2 schedule, no early stop unless a wall justifies it) |
| Optimizer | AdamW |

### The question for this run 

| Model | P1 best | P2 behaviour |
|-------|---------|--------------|
| Llama 3.2-3B | 6.68 | WALL at 5.33 in 100 steps, six identical evals |
| Qwen 2.5-14B | 15.05 | broke the wall, descending below 6.72, no floor yet |
| **Llama 3.1-8B** | **11.36** | **the deciding datapoint** |

3B (3B, 128K vocab) walled. Qwen (14B, WakeOverlay-then-LoRA) descends. the 8B sits at 8B / 128K vocab / Llama-family-gradient-masking. its P2 behaviour separates two hypotheses:

- if 8B P2 **walls** like 3B: the LoRA ceiling is a property of the 128K-vocab Llama architecture, and Qwen broke it via the WakeOverlay route, not via scale alone.
- if 8B P2 **descends** like Qwen: the ceiling is scale-gated, and 8B has enough machinery to route past the friction that stops 3B.

read through μp → UP: P1 laid down the 8B's μ-units (the compositional-init Wake embeddings, frozen now). P2 is the arrow so the question is whether the arrow finds enough headroom to route the μ-units, or saturates against the friction the way 3B's arrow did. 

first eval at step 100. the 3B walled by step 100. if 8B's val at 100 has dropped well below its P1 best and keeps falling at 150, it is descending. if it drops once and pins, it is walling. EVAL_STEPS=100 means the wall (if it comes) gets documented across ~5 evals before any termination call.
